# Data governance & management notes

This project follows a small set of explicit data-management practices so that
results are reproducible, traceable, and trustworthy, the same standards that
matter when supporting client or enterprise data.

## 1. Layered storage with raw immutability
Data moves through clear layers and is never edited in place:

```
data/raw/         immutable source copy (never modified after ingest)
data/processed/   curated, typed, analysis-ready table (regenerated by code)
reports/          derived outputs: figures, metrics, findings
```

`src/01_ingest.py` writes the raw copy once and prints its SHA-256 hash, so the
exact bytes used in any run can be verified later.

## 2. Lineage (traceability)
Every engineered column in the processed table is traceable to a raw source
column and a documented transformation (see `data/README.md` data dictionary
and the inline comments in `src/02_clean.py`). No column appears without a
documented origin.

## 3. Data quality gates
`src/02_clean.py` ends with hard assertions that fail the build if an
expectation is violated, for example:
- row count must be preserved through cleaning,
- all prices must be positive,
- RAM must fall in a plausible 1-128 GB range,
- PPI must be non-null after parsing.

This turns silent data corruption into a loud, early failure.

## 4. Documentation & reproducibility
- A full data dictionary is maintained in `data/README.md`.
- `requirements.txt` pins the library set; `run_pipeline.py` reproduces every
  output from the raw file with one command.
- A second implementation (`analysis/validate_findings.R`) cross-checks the
  main findings in R, reducing the risk of a single-tool artifact.

## 5. Security & privacy
The dataset contains no personal or sensitive data, only product
specifications and prices, so no anonymization is required. The same layered
approach (raw immutability, documented lineage, access-controlled processed
outputs) is what would extend to governed or access-restricted data in a
production setting.
